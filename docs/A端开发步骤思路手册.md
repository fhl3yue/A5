## 2. 任务目标分析

### 2.1 赛题需求理解

本赛题要求围绕智慧景区导览场景，构建一个集数字人交互、景区知识问答、语音服务、个性化推荐以及后台管理于一体的智能导览系统。结合赛题要求与双人协作实际情况，A 端必须承担系统的“能力底座”作用，即为前端页面提供可调用、可联调、可演示的智能服务接口。

### 2.2 A端目标拆解

为了让项目在有限时间内尽快形成可运行闭环，我将 A 端目标拆分为五类：

1. 基础运行目标  
构建一个可以稳定启动的后端服务工程。

2. 数据服务目标  
建立数据库与景区知识数据的导入机制。

3. 智能问答目标  
支持文本提问、语音提问、知识检索和回答生成。

4. 管理分析目标  
支持后台登录、问答日志查询、满意度记录和数据看板。

5. 联调交付目标  
向 B 端提供清晰、稳定的接口文档和联调说明。

### 2.3 设计原则

为适应比赛节奏、异地协作和最终演示需求，A 端整体遵循以下原则：

- 先搭闭环，再做增强
- 先保证能联调，再追求精细表现
- 优先做本地可运行方案，减少对外部服务依赖
- 所有能力都尽量通过接口暴露，方便 B 端并行开发
- 每一阶段都要有可验证结果，而不是只停留在设计上

## 3. 总体设计思路

### 3.1 架构定位

A 端的本质是“后端服务 + 智能能力中台”。它并不直接承担界面展示，而是负责以下工作：

- 接收前端请求
- 调用知识检索与问答逻辑
- 管理语音识别与语音合成
- 记录用户交互日志
- 产出后台看板数据

因此，在系统中，A 端相当于游客端和管理端共同依赖的服务中心。

### 3.2 技术选型思路

考虑到开发效率、比赛可控性和本地部署便利性，A 端选用以下方案：

- 后端框架：FastAPI
- 数据库：SQLite
- ORM：SQLAlchemy
- 文档数据处理：python-docx、openpyxl
- 语音合成：edge-tts
- 语音识别：faster-whisper
- 数据格式：JSON + REST API

之所以采用上述方案，主要原因如下：

- FastAPI 文档自动生成能力强，便于前后端联调
- SQLite 部署轻量，适合比赛和演示阶段
- Python 数据处理生态成熟，适合快速导入 `.docx` 与 `.xlsx`
- edge-tts 接入简单，适合快速形成数字人播报效果
- faster-whisper 可本地运行，避免完全依赖远程语音识别服务

### 3.3 开发主线

A 端的开发不是并行乱做，而是按下面这条主线推进：

1. 先把项目骨架搭起来
2. 再把数据库和知识数据接进去
3. 再打通文本问答
4. 再加路线推荐和后台统计
5. 再补满意度反馈
6. 再补 TTS 音频返回
7. 最后补 ASR 语音识别与纠偏优化

这样推进的好处是：每做完一层，就可以立刻验证，不容易后期整体返工。

## 4. 分阶段实施步骤

## 4.1 第一阶段：初始化 A端工程

### 4.1.1 目标

在 `D:\software` 下建立一个独立的 A 端开发工作区，使后续所有脚本、数据、接口和文档都有统一位置，避免与其他项目混淆。

### 4.1.2 实施内容

- 创建 `backend`、`scripts`、`data`、`docs` 等目录
- 建立 FastAPI 项目骨架
- 配置依赖文件 `requirements.txt`
- 配置环境变量模板 `.env.example`
- 整理一键启动脚本和基础说明文件

### 4.1.3 这样做的原因

如果一开始没有统一目录结构，后面随着接口增多、脚本增多、数据文件增多，很容易出现：

- 启动路径混乱
- 数据文件找不到
- 团队成员无法快速接手
- 提交给老师或答辩时材料不成体系

因此，工程化结构搭建是 A 端的第一步，而不是附属工作。

### 4.1.4 阶段结果

形成了可独立启动的 A 端工作区，主要目录如下：

```text
D:\software
├─ backend
├─ scripts
├─ data
├─ docs
├─ start_backend.ps1
└─ reset_demo_env.ps1
```

## 4.2 第二阶段：数据库与基础数据模型设计

### 4.2.1 目标

让后端具备基本的数据持久化能力，并能够支撑问答日志、后台统计和知识管理等功能。

### 4.2.2 核心数据对象

设计了以下核心表：

- 管理员表 `admin_users`
- 知识文档表 `knowledge_documents`
- 知识片段表 `knowledge_chunks`
- 景点信息表 `scenic_spots`
- 路线预设表 `route_presets`
- 问答日志表 `qa_logs`

### 4.2.3 设计思考

这里没有一开始就做很复杂的多表关系，而是采用“够用、清晰、易扩展”的思路：

- 先保证关键业务能落表
- 再考虑后续是否要拆更细的实体
- 尽量让日志、看板、问答引用都能直接从现有表结构中获得

这样设计的优点是：

- 开发速度快
- 便于自测
- 便于讲解
- 便于后续由 SQLite 平滑升级到 MySQL 等数据库

### 4.2.4 阶段结果

实现了数据库初始化脚本与管理员账号自动创建脚本，支持一键生成初始数据库环境。

## 4.3 第三阶段：景区知识库建设

### 4.3.1 目标

让系统具备回答景区问题的基础知识来源，而不是只返回固定模板。

### 4.3.2 数据来源

本阶段主要处理以下两类数据：

- 官方资料包中的 `.docx` 文档
- 官方资料包中的 `.xlsx` 数据表

同时，为了方便本地快速演示，又手工整理了一套精简样例数据，包括：

- 核心景点样例数据
- 典型游览路线样例数据

### 4.3.3 实施步骤

1. 读取原始文档
2. 清洗无关内容
3. 将内容拆成可检索片段
4. 写入知识片段表
5. 对关键景点额外保存结构化字段
6. 导入路线预设数据

### 4.3.4 设计思考

我没有一开始就上复杂向量数据库，而是先做“可运行的检索增强版本”，原因有三点：

- 比赛阶段首先要保证系统可用
- 现有官方资料体量并不大，先做轻量检索能更快闭环
- 后续如需增强，可以平滑升级为更强的 RAG 实现

### 4.3.5 阶段结果

系统已经可以导入：

- 样例景点数据
- 样例路线数据
- 官方 `.docx` 资料
- 官方 `.xlsx` 资料

并能将这些数据用于后续问答与推荐。

## 4.4 第四阶段：文本问答接口开发

### 4.4.1 目标

实现游客端最核心的文本问答能力，支持：

- 接收问题
- 检索相关知识
- 生成回答
- 返回参考来源
- 记录问答日志

### 4.4.2 实施步骤

1. 设计 `POST /api/chat/text` 接口
2. 编写问题接收与参数校验逻辑
3. 从知识片段中进行相关内容检索
4. 根据命中的景点或片段组织回答
5. 生成音频播放地址
6. 写入问答日志
7. 将结果统一返回给前端

### 4.4.3 设计思考

文本问答是整个系统的基础，因为：

- 语音问答最终也会落到文本问答逻辑
- 后台统计、满意度记录也依赖问答日志
- 如果文本问答都不稳定，后续其他模块都没有意义

因此，我优先把文本问答做成统一入口，再让其他能力围绕它扩展。

### 4.4.4 阶段结果

文本问答已经能够返回：

- 原始问题
- 系统回答
- 音频地址
- 参考来源
- 响应耗时
- 日志编号

这使得前端页面和后台统计都有了统一的数据来源。

## 4.5 第五阶段：路线推荐接口开发

### 4.5.1 目标

根据游客兴趣和游览时长，返回合适的游览路线。

### 4.5.2 实施步骤

1. 设计路线预设数据结构
2. 在数据库中写入历史文化、佛教文化、自然风光、亲子休闲等预设路线
3. 编写 `POST /api/recommend/route` 接口
4. 根据兴趣和时长匹配路线
5. 返回路线名称、景点顺序和推荐理由

### 4.5.3 设计思考

在比赛初期，我没有采用复杂动态规划或实时推荐算法，而是先基于“规则 + 预设路线”做推荐，主要原因是：

- 解释性强，便于老师理解
- 返回结果稳定，适合演示
- 适合作为后续智能推荐的第一版基础

### 4.5.4 阶段结果

路线推荐接口已经稳定可用，可为前端直接提供路线卡片展示数据。

## 4.6 第六阶段：后台登录、日志与数据看板开发

### 4.6.1 目标

为管理端提供登录、问答日志查询和运营数据展示能力。

### 4.6.2 实施内容

- 开发 `POST /api/admin/login`
- 开发 `GET /api/admin/logs`
- 开发 `GET /api/admin/dashboard`
- 开发满意度反馈接口 `POST /api/feedback`

### 4.6.3 数据看板指标

当前看板主要输出以下指标：

- 今日服务人数
- 今日问答次数
- 满意度比例
- 热门问题统计
- 情绪分布统计

### 4.6.4 设计思考

数据看板的作用不仅是“展示数据”，更重要的是作为前端管理页面联调的数据源。因此在设计时，我重点考虑的是：

- 字段简洁稳定
- 统计结果易于前端图表组件消费
- 结果适合演示时快速讲解

### 4.6.5 阶段结果

后台接口已经形成闭环：

- 登录可用
- 日志可查
- 满意度可写入
- 看板可展示

## 4.7 第七阶段：语音播报功能开发

### 4.7.1 目标

让文本问答和语音问答在返回文本答案的同时，还能够返回可直接播放的语音链接，提升“数字人导览”效果。

### 4.7.2 实施步骤

1. 集成 `edge-tts`
2. 在问答完成后自动合成语音
3. 将音频输出到本地目录
4. 用静态资源方式挂载音频目录
5. 返回 `audio_url` 给前端

### 4.7.3 设计思考

语音合成放在这个阶段做，而不是一开始做，原因是：

- TTS 不解决“能不能回答”，只增强“怎么展示”
- 先把问答做通，再补语音输出，更符合开发节奏
- 语音返回后，B 端就可以较容易实现数字人播报界面

### 4.7.4 阶段结果

问答接口已经能够返回可访问的音频地址，前端可以直接用来播放。

## 4.8 第八阶段：语音识别与语音问答开发

### 4.8.1 目标

实现游客上传音频后，系统能够完成：

- 语音识别
- 问题提炼
- 知识问答
- 音频回复

### 4.8.2 技术方案

采用本地 `faster-whisper` 作为语音识别方案，理由是：

- 可离线运行
- 不依赖必须在线的外部 ASR 服务
- 适合比赛演示环境

### 4.8.3 实施步骤

1. 设计 `POST /api/chat/voice`
2. 支持上传音频文件
3. 将音频保存到临时目录
4. 调用 `faster-whisper` 识别文本
5. 提取原始识别结果 `transcript`
6. 对长段识别文本进一步纠偏，提炼为 `interpreted_question`
7. 将提炼后的问题送入文本问答主逻辑
8. 返回最终答案和音频地址

### 4.8.4 设计思考

在实际测试中，我发现语音识别结果并不总是标准问题句，尤其当输入音频较长、包含很多说明内容时，可能识别成一大段文本。为了解决这个问题，我增加了“语音问题纠偏”逻辑：

- 保留原始识别结果 `transcript`
- 同时生成系统理解后的问题 `interpreted_question`

这样做的好处是：

- 前端可以向用户展示“识别到了什么”
- 同时还能展示“系统实际理解成了什么问题”
- 即使识别文本较长，也能尽量保证检索不被带偏

### 4.8.5 阶段结果

语音问答接口已经具备实际可用性，能够：

- 识别上传音频
- 返回原始 transcript
- 返回 interpreted_question
- 返回答案与音频链接

## 4.9 第九阶段：演示环境重置与联调支撑

### 4.9.1 目标

保证每次答辩、演示或联调前，都能快速恢复到一个干净、稳定的环境状态。

### 4.9.2 实施内容

开发了：

- 一键启动脚本
- 演示环境重置脚本
- 全量重置脚本

### 4.9.3 作用

这些脚本能够完成：

- 清空问答日志
- 清空已生成音频
- 重建数据库
- 重建管理员账号
- 重新导入样例与官方资料

### 4.9.4 设计思考

很多比赛项目在答辩时不是功能不足，而是环境脏、测试数据乱、历史记录影响展示效果。因此我把“环境重置”单独作为一个阶段来做，而不是临时清数据库。

### 4.9.5 阶段结果

目前已经可以在演示前通过脚本快速恢复干净环境，提升答辩稳定性。

## 5. 自测与自纠过程

### 5.1 自测原则

我在每完成一个阶段后，都坚持进行可复现的接口自测，而不是等所有代码写完后再统一排错。

### 5.2 主要自测方式

- Swagger 页面直接调用
- `TestClient` 本地脚本调用
- 真实音频文件上传测试
- 演示环境重置后复测

### 5.3 典型自纠内容

在开发过程中，我主动发现并修正了以下问题：

1. 数据库相对路径导致脚本运行目录变化时无法打开数据库  
修正为绝对路径解析方式。

2. 终端中文编码显示混乱  
通过 UTF-8 方式处理脚本输出，减少乱码影响。

3. 看板热门问题混入历史脏长句  
增加过滤规则，只保留适合作为热门问题展示的问句。

4. 语音识别长段文本容易带偏检索  
增加 `interpreted_question` 逻辑，将长文本提炼为更合理的问题句。

5. 知识引用不稳定、主条目不优先  
优化引用排序规则，使景点主条目尽量优先展示。

6. 响应耗时统计不完整  
修正 `response_seconds` 计算时机，使其覆盖完整处理链路。

### 5.4 自纠意义

这些自纠工作虽然不一定直接增加“新功能”，但对项目最终效果影响很大，尤其是在：

- 前后端联调
- 评审展示
- 教师审阅
- 答辩稳定性

等方面具有明显价值。

## 7. 当前不足与后续优化方向

虽然 A 端已经形成闭环，但仍有进一步优化空间：

### 7.1 当前不足

- 本地语音识别在景区专有名词上仍有一定误差
- 文本检索目前为轻量实现，后续可进一步增强为更强的 RAG
- 管理端看板指标仍可扩展
- 登录流程目前为演示版 token，后续可增强为更标准的认证机制

### 7.2 后续优化方向

1. 提升语音识别精度  
根据机器性能选择更合适的 ASR 模型，或引入更强的专有名词提示。

2. 强化知识检索  
加入更成熟的向量检索与重排序机制，提升问答准确率。

3. 丰富后台分析  
增加时间趋势、游客兴趣偏好、满意度对比等分析项。

4. 提升数字人联动效果

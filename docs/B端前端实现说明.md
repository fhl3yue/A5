# 景区导览服务AI数字人项目
# B端前端实现说明

## 1. 文档目的

由于 B 端成员暂时无法继续开发，本阶段对 B 端工作进行了接手补充，目标是尽快形成一个可演示、可联调、可提交的前端版本，使游客端、管理端与 A 端后端接口串联起来。当前页面定位为“通用景区导览 AI 数字人平台”，以灵山胜境作为示范景区进行展示。

## 2. 当前实现方式

当前 B 端采用轻量静态前端方式实现，前端文件位于：

- `D:\software\frontend\index.html`
- `D:\software\frontend\styles.css`
- `D:\software\frontend\app.js`

该方案不依赖额外的 Node.js、Vue、React 或 Vite 环境，直接由 FastAPI 后端托管。启动后端后即可访问：

`http://127.0.0.1:8000/app/`

这样做的主要原因是降低联调复杂度，提升演示和发行版打包的稳定性。

## 5. 前后端串联流程

### 5.1 文本问答流程

1. 用户在游客端输入问题
2. 前端调用 `/api/chat/text`
3. 后端返回答案、音频地址、参考来源和日志 ID
4. 前端显示回答并尝试播放音频
5. 用户提交满意度
6. 前端调用 `/api/feedback`
7. 后台看板数据随日志和评分变化

### 5.2 语音问答流程

1. 用户点击录音或上传音频文件
2. 前端以 `multipart/form-data` 调用 `/api/chat/voice`
3. 后端完成语音识别、问题提炼、知识库问答和语音合成
4. 前端展示 `transcript` 和 `interpreted_question`
5. 前端展示答案并播放 `audio_url`

### 5.3 路线推荐流程

1. 用户选择兴趣偏好和游览时长
2. 用户也可以选择自定义兴趣或自定义时长
3. 前端调用 `/api/recommend/route`
4. 后端根据预设项或自定义输入进行偏好匹配
5. 后端返回路线名称、景点顺序和推荐理由
6. 前端以路线卡片形式展示

### 5.4 管理端流程

1. 管理员使用 `admin / admin123` 登录
2. 前端调用 `/api/admin/login`
3. 登录成功后加载 `/api/admin/dashboard`
4. 前端展示看板指标、热门问题和情绪分布
5. 前端调用 `/api/admin/logs` 展示问答日志
6. 管理员可通过 `/api/admin/docs/upload` 上传知识文档

### 5.5 数字人形象管理流程

1. 管理员进入管理端数字人形象管理区域
2. 设置数字人名称、角色身份和示范景区
3. 选择服装主题，包括灵山青绿、文旅金、湖畔蓝、节庆红
4. 选择语音声线，包括晓晓、云希、晓伊、云健等中文声线
5. 修改欢迎语
6. 点击保存后调用 `/api/admin/digital-human/config`
7. 游客端通过 `/api/digital-human/config` 读取配置并更新角色展示
8. 后端语音合成会使用保存后的声线配置

页面表达上已经区分“系统能力”和“示范景区”：系统是面向景区导览场景的通用平台，当前演示数据和推荐问题来自灵山胜境知识库。

## 8. 当前不足

当前前端属于参赛演示版，仍存在以下可优化空间：

1. 数字人形象目前为原创 SVG/CSS 类 Live2D 分层动效角色，不是 3D 数字人，也不是 Live2D Cubism SDK 模型
2. 图表为轻量 CSS 展示，没有接入 ECharts
3. 管理端登录为演示 token，没有完整权限体系
4. 景区配置目前以展示和知识文档上传为主，还没有完整的多景区切换表单
5. 项目不内置初音未来等第三方虚拟偶像素材，授权角色模式仅用于接入用户已获得授权的本地文件
5. 语音录制效果受浏览器麦克风权限和本机环境影响
6. 页面已经可用，但仍可进一步打磨视觉细节和演示文案
